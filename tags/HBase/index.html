深入理解HBase架构（翻译）

原文地址

HBase架构

从物理结构上说，HBase采用主从结构，主要由三种类型的服务组成。Region servers提供数据的读写服务。当获取到数据时，客户端直接与HBase RegionServers通信。Region的分配、DDL（创建、删除表）操作都交由HBase Master进程处理。Zookeeper，作为HDFS的一部分，主要负责维护集群的状态。

Hadoop DataNode存储了Region Server管理的数据。所有HBase的数据都被存储在HDFS files中。Region Servers与HDFS DataNodes搭配，保证了RegionServers提供的数据的位置（把数据放在需要的地方）。HBase数据在写入时是本地数据，但当region被移动时，在被压缩前不是本地数据。

NameNode为所有物理数据块包含的文件维护元数据的信息

Regions

HBase表被row key水平地切分为一个又一个的”region”。一个region包含了其start key至end key范围内HBase表中的所有行。当多个region被分配到集群中的节点时，就被称为“Region Servers”，其提供了数据的读写服务。一个region server可以服务大约1000个region。

HBase HMaster

Region分配，DDL（创建、删除表）操作都由HBase Master处理。

一个Master负责：

  • 协调region servers
    • 在启动时分配region，重新分配region以进行恢复或负载均衡
    • 监控所有在集群中的RegionServer实例（监听来自zookeeper的通知）
  • 管理功能
    • 提供创建、删除、更新表的接口

ZooKeeper：协调员

HBase使用ZooKeeper作为分布式协调服务去维护集群中的服务器状态。Zookeeper维护着处于活动状态的服务器，并提供服务器故障通知。ZooKeeper使用consensus去保证共同的共享状态。需要注意的是，至少需要三台或五台机器达成consensus。

各个组件是如何一起工作的

Zookeeper被用作协调分布式系统中各个成员的共享状态信息。所有Region Server和活跃的HMaster通过session与ZooKeeper连接。ZooKeeper通过心跳机制维持与临时节点之间的活动会话。

每个Region Server创建一个临时节点。HMaster通过监控这些节点去发现所有可用的region server，同时也监控这些节点的服务器故障。每个HMaster也会争相创建一个临时节点。Zookeeper确定第一个然后使用它来确保只有一个master是活跃的。活跃的HMaster发送心跳给到Zookeeper，然后待用的HMaster监听活跃HMaster失败的通知。

如果一个region server或者活跃的HMaster发送心跳失败，那么session就会过期并且相应的临时节点会被删除。删除节点的信息会通知到更新监听器。活跃的HMaster监听region server，然后会恢复失败的region server。备用的HMaster则会监听活跃的HMaster失败信息，如果活跃的HMaster挂了，那么备用的HMaster就会转为活跃状态。

HBase首次读或写

有一个特殊的HBase目录表被称为META表，此表保存了所有region在集群中的位置。Zookeeper储存META表的位置。

这是客户端第一次读或写入HBase时发生的情况：

1：客户端从ZooKeeper获取承载META表的Region server
2：客户端将会查询.META.服务获取与其想要访问的row key相对应的region server。客户端将这条信息与META表位置一起缓存
3：客户端将会从对应的Region Server获取Row

对于未来的读操作，客户端使用缓存来取回META的位置和先前读取的行键。随着时间的推移，不需要读取META表，除非一个region被移动导致的查询失败。如果是这样的话它将会重新查询并更新缓存。

HBase Meta表

  • META表是一张保存着系统中所有region目录的表
  • .META.表像B Tree
  • .META.表结构如下：
    • Key: region start key，region的id
    • Values: RegionServer

Region Server的组件

一个Region Server在HDFS数据节点上运行，并且有着如下组件：

  • WAL： Write Ahead Log是分布式文件系统中的一个文件。WAL被用作存储那些尚未被持久化到永久存储上的新数据。它常常被用作发生故障时的恢复。
  • BlockCahce： 这是个读缓存。它将频繁读取的数据放置于缓存之中。当缓存满了时最近没使用的数据会被清除。
  • MemStore： 这是个写缓存。它保存着那些还没有被写入磁盘的新数据。在写入磁盘前进行排序。每一个region中每一个列族都会有一个MemStore。
  • Hfile将行数据存储为磁盘上排序好的Key Values。

HBase写步骤 [1]

当一个客户端发出Put请求时，第一步是将数据写入预写入日志WAL:

  • 编辑好的文件将会被追加到位于磁盘上的WAL文件末尾
  • WAL用于在服务器奔溃是恢复尚未保存的数据

HBase写步骤 [2]

当数据写入至WAL后，就会被放入MemStore。然后，put请求确认就会返回给客户端。

HBase MemStore

MemStore把内存中的更新保存为排序好的键值对，与保存在HFile中的相同。每个列族中都会有一个MemStore。更新时按照每一个列族排序。

HBase Region刷新

当MemStore积累到足够的数据时，整个有序集合就被写入到HDFS中的新HFile中。HBase中每个列族使用多个HFile，其中包含了实际的单元格，或是键值对实例。当在MemStore里排序好的键值对编辑文件被刷新后作为文件保存到磁盘时，这些HFile文件就会随着时间的推移而创建。

需要注意的是，这是HBase中列族数量被限制的一个原因。在每一个列族中都会有一个MemStore；当一个满了，所有的都会刷新。它同时还保存了最后写入的序列号，以便系统知道目前为止被持久化的数据。

最大的序列号作为每个HFile中的元数据字段存储，来反映哪里结束哪里开始。当一个region启动时，这个序列号就会被读取，同时最大的数作为新编辑文件的序列号。

HBase HFile

HFile中保存着键值对数据。当MemStore积累到足够的数据时，整个被排序好的键值对集合会被写入到HDFS上的HFile中。这就是顺序写入。这一过程非常快因为它避免了移动磁盘头。

HBase HFile结构

一个HFile中包含的多层索引允许HBase在不用读取所有文件的情况下查找数据。多级索引就像一个B+树：

  • 键值对按递增顺序存储

  • 索引按照行键指向64KB块中的键值数据

  • 每个块都有自己的叶索引

  • 每个块最后的一个键放在中间索引中

  • 根索引指向中间索引

尾部块指向元数据块，并且被写入到文件中的持久化数据的末尾。尾部块同时也拥有着比如bloom过滤器和时间范围等信息。Bloom过滤器有助于跳过不包含某个确定行键的文件。如果文件不在读取的时间范围内，则时间范围对跳过文件非常有用。

HFile 索引

我们之前所讨论的索引，当HFile被编辑时加载并保存在内存中。这允许使用单个磁盘查找进行搜索。

HBase读合并

我们可以看到，对应一行的KeyValue单元格可以在多个位置，单元格在已经持久化的HFiles中，最近更新的单元格在MemStore中，最近读的单元格在块内存中。所以当你读取一行时，系统如何知道获取对应的单元格并返回？在如下步骤中，read合并块内存，MemStore\和HFiles中的键值：

1. 首先，扫描器查找在块缓存中的行单元格 - 读缓存。最近读取的键值被缓存在这，当需要内存时，最近不经常使用的键值就会被回收。
2. 接下来，扫描器就会在MemStore中查找最近写入内存的写缓存。
3. 如果扫描器在MemStore和块缓存中没有找到所有的行单元格，那么HBase将会使用块缓存索引和bloom过滤器把HFiles加载到内存中，该文件可能含有目标单元格。

就像之前讨论的，在每个MemStore中可能有很多HFiles，这意味着对于读取，可能必须检查多个文件，这可能会影响性能。这被称为读取放大。

HBase轻压缩

HBase会自动挑选一些大小较小的HFile，把它们重写到一个更大的HFile中，这个过程被叫做轻压缩。通过重写多个小文件到数量更少体积更大的大文件中，轻压缩执行合并排序减少了存储文件的数量。

HBase重压缩

重压缩合并和重写region中的所有HFile，并将其写入至每一个列族中的HFile中。在这一过程中，会丢弃已删除或已过期的单元格。这一举动提升了读性能。然而，在重压缩重写所有文件时，可能会引发很多IO和网络的问题。

重压缩可以被安排为自动运行。由于写入可能引发的问题，所以其一般安排在周末或者晚上。需要注意的是，MapR-DB已经进行了改进，不需要进行压缩。一个重压缩还会使任何因为服务器故障或者负载均衡而产生的远程文件，变为region文件。

Region = 邻接键

让我们快速复习下region：

  • 一个表可以水平地分成一个或者多个region。一个region包含起始键和结束键之间的相邻排序行数据。

  • 每一个region默认1GB大小

  • 表的一个region由RegionServer提供服务给客户端

  • 一个region server可以服务大约1000个region（可能同属于一张表或不同的表）

Region 切分

在最初始的时候，每个表都会有一个region。当一个region数据增长过大时，这个region就会切分成2个子region。两个子region，各代表原始region的一半，在同一个region server上并行打开，然后切分情况会被报告给HMaster。出于负载均衡的原因，HMaster也许会将新region移动到其他服务器。

负载均衡

拆分最初发生在同一个region服务器上，但出于负载均衡的原因，HMaster也许会将新region移动到其他服务器。这导致新的Region server从远程HDFS节点提供数据。直到主压缩把数据文件移动到Regions server的本地节点。HBase数据在写入时是本地数据，但是当一个region被移动时（用于负载均衡或者恢复数据），这些数据在主压缩完成之前都不会是本地数据。

HDFS数据复制

所有写入和读取都是来自/去到主节点。HDFS复制WAL和HFile块。HFile块复制自动发生。当这些块文件存储好后HBase依赖于HDFS去提供数据安全。当数据写入到HDFS后，一份备份会写入到本地，然后它会被复制到第二个节点，再之后第三个备份会写入到第三个节点。

WAL文件和HFiles都被持久化到磁盘和或者备份，那HBase如何恢复没有持久化到HFiles中的MemStore更新呢？下个章节会看到答案。

HBase损坏修复

当一个region server故障时，在检查和修复程序启动前损坏的Region都不可用。Zookeeper检测不到region server的心跳后就会定位失败的节点。在这之后，HMaster就会被通知Region Server已挂。

当HMaster监测到挂掉的region server后，HMaster就会把region从挂掉的region server分配到活跃的region server。为了恢复挂掉的region server中的memstore里的没有刷新至磁盘的编辑文件，HMaster把属于挂掉的region server里面的WAL文件拆分到不同的文件中，然后把这些文件存储到新的region server数据节点之中。然后每个region server把各个分散的WAL文件复原，并修复region的memstore。

数据恢复

WAL包含了编辑文件的列表，同时一个编辑文件代表了一个简单的添加或删除。编辑文件按时间顺序写入，所以当需要持久化时，新增内容就被添加到保存在磁盘里的WAL文件的末尾。

当数据仍旧在内存中或是没有持久化到内存时，如果此时服务挂掉会发什么？WAL文件就会重新生成。读取指定的WAL文件就可以生成一个新的WAL文件，把持有的编辑文件排序并添加到当前的memstore中。在最后，Memstore就会刷新并把变化内容写入至HFile中。

HBase架构的优点

HBase提供了如下好处：

  • 强一致性模型
    • 当写成功后，所有reader都会看到一样的值
  • 动态伸缩
    • 当数据过大时Region会拆分
    • 使用HDFS去传输和复制数据
  • 内置恢复
    • 使用提前写入日志
  • 与Hadoop集成
    • HBase上的MapReduce非常简单

HBase的缺点

  • 业务连续可靠性
    • WAL重新生成缓慢
    • 崩溃恢复缓慢复杂
    • 大量IO流
