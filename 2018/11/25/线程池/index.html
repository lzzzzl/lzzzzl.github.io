线程池

线程池是啥

线程池是一种多线程处理形式，处理过程中将任务添加到队列，然后在创建线程后自动启动这些任务。线程池可以避免系统频繁地创建和销毁线程，以及让创建的线程可以进行复用。

线程池中的一些方法

  • newFixedThreadPool()：该方法返回一个固定线程数量的线程池，线程池中的线程数量始终不变。当有一个新的任务提交时，线程池中若有空闲的线程，则立即执行。若没有，则新的任务会被暂存在一个任务队列中，待有线程空闲时，便处理任务队列中的数据。

  • newSingleThreadExecutor()：该方法返回一个只有一个线程的线程池。若多余一个任务被提交到线程池，那么任务会被保存在一个任务队列中，待线程空闲时，按照先进先出的顺序执行队列中的任务

  • newCacheThreadPool()：该方法返回一个可根据时间情况调整线程数量的线程池。线程池的线程数量不确定，但若有空闲线程可以复用，则会优先使用可复用的线程。若所有线程均在工作，此时又有新的任务提交，则会创建新的线程处理任务。所有线程在当前任务执行完毕之后，将返回线程池进行复用。

  • newSingleThreadSchedulerExecutor()：该方法返回一个SchedulerExecutorService对象，线程池大小为1。SchedulerExecutorService接口在ExcutorService接口上扩展了在给定时间执行某任务的功能，如在固定时间的延时之后执行，或者周期性执行某个任务。

  • newSchedulerThreadPool()：该方法也返回一个SchedulerExecutorService对象，但该线程池可以指定线程数量。

当采用scheduleAtFixedRate()时，如果任务的执行时间超过调度时间，此时并不会出现任务堆叠的情况，而是任务的调度时间变为执行时间。如果采用的是schedulerWithFixedDelay()，那么此时的任务的时间间隔时间为则为调度时间+任务时间。

线程池扩展

ThreadPoolExecutor提供beforeExecutor()、afterExecutor()、teriminated()三个接口对线程池进行控制。

使用shutdown()方法关闭线程池，它并不会立即暴力地终止所有任务，它会等待所有线程执行完成后再返回。

线程池超负载问题优化：拒绝策略

当任务数量超过系统实际承载时，需要用到ThreadPoolExecutor的拒绝策略。

  • AbortPolicy策略：该策略会直接抛出异常，阻止系统工作。
  • CallerRunsPolicy策略：只要线程池未关闭，该策略直接在调用者线程中，丢弃当前被运行的任务。此时，任务提交线程的性能可能会急剧下降。
  • DiscardOledestPolicy策略：该策略将丢弃最老的一个请求，也就是即将执行的一个任务，并尝试再次提交任务。
  • DiscardPolicy策略：该策略默默丢弃无法处理的任务，不予任何处理。

ForkandJoin框架

fork()后系统多了一个执行分支（线程），所以需要等待这个执行分支执行完毕，才有可能得到最终的结果。因为毫无顾忌地使用fork()会导致系统开启过多的线程而导致严重影响性能。所以，JDK给出了一个ForkJoinPool线程池，来处理fork()

线程池大小公式

  • Ncpu = CPU的数量
  • Ucpu = 目标CPU的使用率，0 <= Ucpu <= 1
  • W/C = 等待时间与计算时间的比率

最优池的大小为：
Nthreads = Ncpu Ucpu (1 + W/C)

一些好用的工具类

  • ConcurrentLinkedQueue：高并发环境中性能最好的队列
  • CopyOnWriteArrayList：将读性能发挥到极致，读取时完全不加锁，写入不会阻塞读取操作，只有写入与写入之间需要进行同步等待
  • BlockingQueue：是一个数据共享通道，当我们希望线程A能够通知线程B的，又不希望线程A知道线程B的存在时，就可以使用BlockingQueue。BlockingQueue同时可以作为消费者和生产者的解耦
    • ArrayBlockingQueue：基于数组实现，适合做有界队列
    • LinkedBlockingQueue：基于链表实现，适合做无界队列，或者边界值非常大的队列
  • SkipList：跳表，是一种可以用来作快速查找的数据结构，有点类似于平衡树。不同之处是，对平衡树的插入和删除往往可能导致平衡树进行一次全局调整。而对跳表的插入和删除只需要对整个数据结构的局部进行操作即可。在高并发环境下，使用跳表可以获得更好的性能。
